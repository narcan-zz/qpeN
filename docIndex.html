generateKeys API - This module accepts a username and a key size. It generates 2 files with the given username as the file name; a public key file with a '.pbk' extension and a private key file with a '.pvk' extension. Both files contain numerical keys of the size specified. The 'username.pbk' file is the public key consisting of 'e' followed by a space, followed by 'N'. The 'username.pvk' file is the private key consisting of 'd' followed by a space, followed by 'N'.

The two files are created (or replace exisiting files by the same name) in the current working directory (CWD). This arrangement is comparable to saying that the keys for an example input are:

    For an input "Alice 1024", the resulting files would contain:
  • The public key: [e,N] as "e N" in file 'Alice.pbk' where e and N are 1024 bit numbers
  • The private key: [d, N] as "d N" in file 'Alice.pvk' where d and N are 1024 bit numbers
encipher API - This module accepts 3 file names. The first file is the ASCII text file with the message to be enciphered. The second file must be the public key file to be used to encipher the plaintext. The third file must be the private key file to be used in signing the enciphered plaintext.

It generates a 'plaintextFile.txt.cph' file containing the result of enciphering the plaintext in the input file ('plaintext.txt').

The file is created (or replaces an exisiting file by the same name) in the current working directory (CWD). An example if Alice wants to encipher a message to Bob so that he's sure that she sent it (Note: she must first obtain Bob's public key, either in a directory or by having him send it to her):

    For an input "plaintext.txt Bob.pbk Alice.pvk", the result will be:
  • An enciphered file 'plaintext.txt.cph'
  • It will contain the result of enciphering 'plaintext.txt' with 'Bob.pbk' and signing it with 'Alice.pvk'
decipher API - This module accepts 3 file names. The first file is the enciphered file containing a string of numbers and spaces. The second file must be the public key file to be used to authenticate the author of the ciphertext. The third file must be the private key file to be used in deciphering the authenticated plaintext. It returns a text file with the ascii characters of the result of translating the deciphered file with a '.txt' extension. If all goes well, it will match the original message exactly.

The resulting 'plaintextFile.txt.cph.txt' file is the file containing the result of deciphering the ciphertext in the input file ('plaintext.txt.cph').

The file is created (or replaces an exisiting file by the same name) in the current working directory (CWD). An example if Bob wants to decipher an enciphered message which is supposedly from Alice (Note: he must first obtain Alice's public key, either in a directory or by having her send it to him):

    For an input "plaintext.txt.cph Alice.pbk Bob.pvk", the result will be:
  • A deciphered file 'plaintext.txt.cph.txt'
  • It will contain the result of authenticating 'plaintext.txt.cph' with 'Alice.pbk' and deciphering it with 'Bob.pvk'
